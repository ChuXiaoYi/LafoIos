# Lafo 一期产品设计规格

日期：2026-05-10
产品：Lafo / 拉否 / 今天拉了吗
平台：iOS 原生 App
首版定位：可爱、有趣、低负担的排便打卡 App

## Objective

Lafo 要让普通成年人用 10 秒完成排便打卡，轻松记录每天排便次数、状态、时间和肠道趋势。产品不是严肃医疗工具，而是一个带陪伴感和分享欲的生活记录 App。

首版核心体验：

- 打开 App，看到“今天拉了吗？”
- 点“打卡”，用 emoji 选择这次状态。
- 保存后看到“此刻也有 X 人在拉屎”。
- 可以生成一张可爱的 Lafo 分享卡。
- 首页沉淀月度趋势日历和今日次数。

## Product / PRD

### Background

排便记录本身私密、尴尬、容易被用户放弃。Lafo 用“拉否”的名字梗、`🚽 + 💩` 的 emoji 表达和轻量打卡流程，把记录行为从严肃表单变成每天顺手完成的小仪式。

### Target Users

首版目标用户是普通成年人，尤其是愿意做轻量生活记录、喜欢有趣表达、但不想被健康 App 教育或诊断的用户。

### User Problem

用户想知道自己最近排便是否规律，但传统记录方式麻烦、严肃、缺少反馈；排便话题又有天然分享梗，用户愿意用轻松方式表达“今天拉了吗”。

### Scope

一期必须包含：

- iOS 原生 App。
- Apple 登录和手机号验证码登录。
- 首页今日摘要和月度趋势日历。
- 打卡记录：时间、emoji 状态、可选备注。
- 一天可多次排便打卡。
- “没拉”作为主动打卡状态，区别于未记录。
- 后端同步用户记录。
- 后端返回半真实氛围数：`此刻也有 X 人在拉屎`。
- 分享卡在 iOS 端生成。
- 分享卡文案和同拉人数参数支持后端配置。

### Non-goals

一期不做：

- 社交广场。
- 好友、关注、评论、排行榜。
- 陌生人动态、头像、昵称或位置展示。
- 医疗诊断、疾病判断、治疗建议。
- 开始/结束计时。
- 宠物养成和成就贴纸系统。

### User Flow

1. 用户首次打开 Lafo。
2. 用户用 Apple 登录或手机号验证码登录。
3. 用户进入首页，看到 Lafo、`今天拉了吗？`、今日摘要和趋势日历。
4. 用户点击“打卡”。
5. 用户选择状态：`😌 顺畅`、`😬 有点难`、`💦 有点急`、`🫥 没拉`。
6. 用户可修改时间，可选填一句备注。
7. 用户保存打卡。
8. App 展示打卡成功反馈和同拉人数。
9. 用户可生成分享卡并保存或分享。
10. 首页趋势日历更新。

### Functional Requirements

#### 登录

- 支持 Apple 登录。
- 支持手机号验证码登录。
- 两种登录方式都映射为同一个 Lafo 用户账号。
- 登录只用于同步、恢复和基础安全，不展示社交身份。
- 用户换机后登录同一账号可以恢复历史打卡记录。

#### 首页

- 展示品牌名 Lafo。
- 展示副标题：`今天拉了吗？`
- 展示主按钮：`打卡`
- 展示今日次数。
- 展示一句可爱但不医疗化的趋势反馈。
- 展示月度趋势日历。

#### 打卡

- 默认时间为当前时间，允许用户修改。
- 状态必选。
- 备注可选，限制为一句短文本。
- 同一天可保存多条排便记录。
- `🫥 没拉` 表示用户主动确认当天没有排便，不计入排便次数。

#### 趋势日历

- 未记录日期保持空白。
- 主动选择 `🫥 没拉` 的日期显示为“没拉日”。
- 有排便记录的日期按当天最后一次排便状态展示主状态。
- 今日次数统计只计算排便状态，不计算 `🫥 没拉`。

#### 同拉人数

- 文案主句：`此刻也有 X 人在拉屎`
- 后端返回半真实氛围数。
- 计算参数由后端控制，包括基础人数、时段曲线、扰动范围、最低值、最高值。
- App 不展示真实用户列表、不展示位置、不展示头像昵称。
- 说明文案使用轻松表达：`这是 Lafo 估算出来的陪伴数字，不是孤军奋战。`

#### 分享卡

- iOS 端生成分享卡图片。
- 默认展示 Lafo、`🚽💩`、今日状态、同拉人数和趣味文案。
- 默认不展示备注、精确时间、手机号、账号信息。
- 分享动作使用 iOS 系统分享能力。

### Copy Tone

Lafo 的文案要可爱、短、轻，不要像健康报告，也不要像后台提示。

文案原则：

- 用短句，不写长说明。
- 可用 emoji，但不能堆满屏。
- 语气像朋友提醒，不像医生诊断。
- 可以有梗，但避免低俗、羞辱和脏感。
- 不使用“异常”“疾病”“风险”等医疗判断词。
- 错误和隐私文案也保持温和，但要准确。

可用示例：

- `今天拉了吗？`
- `打卡成功，肠道小宇宙已记录 🚽`
- `此刻也有 18,420 人在拉屎，不是孤军奋战。`
- `今天很顺，给自己一点掌声 😌`
- `没拉也算认真记录 🫥`
- `分享卡已生成，隐私内容不会出现。`

避免示例：

- `您的排便数据已成功提交至服务器。`
- `系统检测到排便异常。`
- `根据您的粪便状态推断肠道风险。`
- `快来和陌生人一起拉屎聊天。`

### Acceptance Criteria

- 新用户可以通过 Apple 登录或手机号验证码登录进入 App。
- 用户能在 10 秒内完成一次标准打卡。
- 用户能保存多条同日排便记录。
- `🫥 没拉` 不计入今日排便次数。
- 未记录日期和主动没拉日期在趋势日历上可区分。
- 打卡成功后展示同拉人数。
- 同拉人数来自后端接口，且不会暴露任何真实用户身份。
- 分享卡默认不包含备注、精确时间、手机号或账号信息。
- 关闭网络时，已登录用户可以在本地完成打卡，网络恢复后同步。
- App 内没有社交广场、评论、好友或排行榜入口。

### Metrics

- D1 留存率。
- 7 日内平均打卡天数。
- 每日打卡完成率。
- 打卡完成耗时。
- 分享卡生成率。
- 分享卡系统分享触发率。
- 登录成功率。
- 同步失败率。
- 用户对“太脏”“太医疗化”“隐私担忧”的反馈量。

## Development / Technical Plan

### Recommended Architecture

一期采用 iOS 原生客户端 + 后端 API。

- iOS：SwiftUI。
- 本地缓存：SwiftData 或 Core Data。
- 后端：REST API 或轻量 JSON API。
- 数据库：PostgreSQL。
- 鉴权：Apple identity token、手机号验证码、Lafo session token。
- 短信：接入短信服务商，并限制验证码发送频率。

- `updatedAt`
- `lastLoginAt`
- `deletedAt`

- `POST /auth/phone/send-code`

- `count`

#### Config

- `GET /config/share-copy`
- `GET /config/app`

### Offline and Sync

- App 本地先写入打卡记录。
- 网络可用时同步到后端。
- 冲突策略：同一条记录以 `updatedAt` 较新的版本为准。
- 删除使用软删除，避免离线删除后又被恢复。
- 同步失败时使用可爱但明确的提示：`网络有点堵，Lafo 稍后再同步。`

### Chinese Comments and Logs Plan

代码中只在非显而易见逻辑处写中文注释：

- `没拉` 与 `未记录` 的区别。
- 同日多次打卡的趋势聚合规则。
- 同拉人数是氛围估算，不代表真实在线用户。
- 分享卡隐私字段过滤。

后端日志使用中文记录重要业务事件：

- 登录成功和失败原因分类。
- 验证码发送受限。
- 打卡同步成功、失败和冲突处理。
- 同拉人数配置读取失败时的兜底。

日志不得包含备注原文、手机号明文、Apple token 或验证码。

### Test Plan

客户端测试：

- 打卡创建、编辑、删除。
- 同日多次打卡统计。
- `🫥 没拉` 不计入次数。
- 未记录与没拉的日历展示。
- 离线打卡和恢复同步。
- 分享卡隐私字段过滤。

后端测试：

- Apple 登录 token 验证流程。
- 手机验证码发送频控。
- 手机验证码校验。
- 记录 CRUD 权限隔离。
- 同步冲突处理。
- 同拉人数区间和扰动稳定性。
- 配置接口兜底。

### Engineering Risks

- 手机号验证码会引入短信成本和风控工作。
- 登录会增加首屏转化摩擦，需要让 Apple 登录成为最顺入口。
- 排便记录属于私密数据，需要严格控制日志和分享字段。
- 同拉人数如果过于真实化，用户会产生隐私误解。

## Operations / Launch Plan

### Audience

首批面向愿意尝鲜、喜欢生活记录和轻松梗表达的 iOS 用户。

### Owner

- Product owner：确认范围、文案尺度和隐私边界。
- Engineering owner：负责 iOS、后端、登录、同步和分享卡。
- Operations owner：负责 TestFlight、种子用户、反馈收集和上架素材。

### Rollout Steps

1. 内部 TestFlight：验证登录、打卡、同步、分享卡。
2. 20-50 人种子测试：验证打卡意愿、文案尺度和分享欲。
3. 小范围公开测试：观察 D1 留存和分享卡生成率。
4. App Store 首发：强调可爱打卡，不强调医疗健康。

### Customer Message

主文案方向：

`Lafo，今天拉了吗？用 10 秒记录你的肠道小日常。打卡后看看此刻还有多少人也在认真生活。`

### Activation Path

- 首次打开先展示一句轻量欢迎文案。
- 登录后直接进入首页。
- 首页主按钮只有一个：`打卡`
- 首次打卡成功后引导生成分享卡。

### Success Metrics

- D1 留存达到可继续迭代标准。
- 新用户首日打卡完成率。
- 分享卡生成率。
- 登录成功率。
- 7 日内至少 3 天打卡的用户占比。

### Guardrail Metrics

- 登录失败率。
- 短信验证码失败率和成本。
- 同步失败率。
- 用户删除账号或卸载反馈。
- 隐私相关负反馈。
- App Store 审核关于医疗或低俗内容的风险反馈。

### Monitoring

- 登录链路成功率。
- 打卡创建接口成功率。
- 同步接口错误率。
- 同拉人数接口错误率。
- 配置接口错误率。
- 分享卡生成失败率。

### Feedback Channels

- TestFlight 反馈。
- App 内反馈入口。
- 种子用户微信群或飞书群。
- 上架后 App Store 评论。

### Support FAQ

- 同拉人数是真实的吗？
  - 这是 Lafo 根据活跃趋势估算的陪伴数字，不展示任何真实用户。
- 我的记录会被别人看到吗？
  - 不会。记录默认私密，分享卡也不会带备注和精确时间。
- 没拉为什么也能打卡？
  - 因为“确认没拉”和“忘记记录”不一样。
- Lafo 会判断我是否健康吗？
  - 不会。Lafo 只做个人记录和趋势回顾，不提供医疗诊断。

### Failure Modes

- 短信验证码不可用：保留 Apple 登录入口，并提示稍后重试。
- 同步失败：本地保存，网络恢复后重试。
- 同拉人数接口失败：使用本地兜底数字和温和文案。
- 分享卡生成失败：允许用户重试。
- 用户担心隐私：在设置和分享页明确说明默认私密。

### Escalation and Rollback

- 如果登录失败率持续升高，临时弱化手机号入口，优先 Apple 登录。
- 如果同拉人数引发隐私误解，文案改为 `Lafo 估算：此刻也有 X 人在拉屎`。
- 如果分享卡被认为低俗，减少 `💩` 出现频率，强化 `🚽` 和 Lafo 品牌。
- 如果 App Store 对医疗表述敏感，移除“肠道趋势”中可能被理解为诊断的描述。

### Review Cadence

- 内测首日：看登录、打卡、同步是否稳定。
- 内测第 3 天：看文案反馈和分享卡使用。
- 内测第 7 天：看留存、打卡天数和核心负反馈。
- 首发后每周：复盘指标并决定是否推进宠物陪伴或成就贴纸。

## Roadmap

二期候选：

- `🚽` 或 `💩` 便便宠物陪伴。
- 连续打卡、贴纸、轻成就。
- 更丰富的趋势视图。
- 数据导出。
- Apple Health 相关能力评估。

二期仍需坚持：

- 不做陌生人社交。
- 不做医疗诊断。
- 不把私密记录变成公开内容。

- 同拉人数语义：`此刻也有 X 人在拉屎`。
